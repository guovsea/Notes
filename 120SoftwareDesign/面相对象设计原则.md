## 分解

将大问题分解为小问题。*对不同的类型的对象使用不同的方法*

```cpp
// 圆
class Circle;
// 矩形
class Rect;
// 画图形
vector<Circle> circleVec;
vector<Rect> rectVec;
for (auto c : circleVec) {
     c.draw();
}
for (auto r : rectVec) {
    r.draw();
}
```

## 抽象

忽略其非本质细节，去处理泛化和理想化了的对象模型。*将具体的类型抽象为基类，再由基类派生出具体的类型*

**抽象使得能够统一处理不同类型的东西**

```cpp
// 抽象基类
class Shape;
// 圆
class Circle : public Shape;
// 矩形
class Rect : public Shape;
// 画图形
vector<Shape*> vec;
for (auto i : vec) {
    i->draw();
}
```

## 面相对象思想

对抗变化，减少变化带来的代码修改

面相对象更强调各个类的责任，各负其责。新增类不应该影响原来类型的实现。

## 面相对象设计原则

### 依赖倒置原则 (DIP)

- 高层模块（稳定）不应依赖于底层模块（变化），应依赖于抽象（稳定）
- 抽象（稳定）不应依赖于实现细节（变化），应依赖于抽象（稳定）

**不依赖**：*指的是不引入某模块就能够独立编译*

### 开放封闭原则（OCP）

- 对扩展开放，对更改封闭
- 类模块应该是可扩展的，但是不可修改

*需求变化时，不要想着去改，而是去试着增加一些东西*

### 单一职责原则（SRP）

- 一个类应该仅有一个引起它变化的原因
- 变化的方向隐含着类的责任

### Liskov 替换原则（LSR）

- 子类必须能够替换它们的基类（IS-A）
- 继承表达类型抽象

### 接口隔离原则（ISP）

- 不应该强迫客户程序依赖它们不用的方法
- 接口应该小而完备

### 组合优于继承

- 类继承通常为“白箱复用”，对象组合通常为“黑箱复用”
- 继承在某种程度上破坏了封装性，子类父类耦合度高
- 而对象组合只要求被组合对象具有良好定义的接口，耦合度低

### 封装变化点

- 使用封装来创建对象之间的分解层，让设计者可以在分界层一侧进行修改，而不会对另一侧产生不良影响，从而实现层次间的松耦合

### 针对接口编程，而不是针对实现编程

*接口：API*

- 不将变量类型声明为某个特定的具体类，而是声明为某个接口 *非绝对*
- 客户程序无需获知对象的具体类型，只需要知道对象所具有的接口
- 减少系统中各个部分的依赖关系，从而实现“高内聚，低耦合”的类型设计方案

*接口标准化是产业强盛的标志。*

## 设计经验（基于设计原则）

1. 设计习语 Design Idioms
   描述**与特定编程语言相关**的底层模式，技巧，惯用法。*如PIMP*
2. 设计模式 Design Patterns
   **类与相互通信的对象**之间的组织关系，包括它们的角色、职责、协作等方面
3. 架构模式 Architectural Patterns
   描述**系统中与基本结构组织**关系密切的高层模式，包括子系统划分，职责，以及如何组织它们之间关系的规则
