**线程是最小的运行单位。**

**进程是最小的资源分配单位。**

同一个进程内多个线程之间可以共享代码段、数据段、打开的文件等资源，但每个线程各自都有一套独立的寄存器和栈，这样可以确保线程的控制流是相对独立的。

进程中，所有线程共享相同的地址空间（虚拟地址和物理地址的映射关系相同）。**线程虽然各自有独立的栈，但是可以通过地址访问其他线程中的数据**。

*这在进程之间是不可能的，因为进程之间不共享地址空间，即使地址相同，所映射到的实际物理地址也不相同。因此必须使用* [[共享内存]]或者 [[mmap]]

## 线程与进程的比较

**不同**

- 进程是资源（包括内存、打开的文件等）分配的单位，线程是 CPU 调度的单位；
- 进程拥有一个完整的资源平台，而线程只独享必不可少的资源，如寄存器和栈；
- 线程同样具有就绪、阻塞、执行三种基本状态，同样具有状态之间的转换关系；
- 线程能减少并发执行的时间和空间开销；

**线程开销较小**

- 线程的**创建**时间比进程快，**因为进程在创建的过程中，还需要管理资源**，比如内存管理信息、文件管理信息，而线程在创建的过程中，不会涉及这些资源管理信息，而是共享它们；
- 线程的**终止**时间比进程快，因为**线程释放的资源相比进程少很多**；
- 同一个进程内的**线程切换**比进程切换快，因为**线程具有相同的地址空间**（虚拟内存共享），这意味着同一个进程的线程都具有同一个页表，那么在切换的时候不需要切换页表。而对于进程之间的切换，切换的时候要把页表给切换掉，而页表的切换过程开销是比较大的；[[上下文切换]]
- 由于**同一进程的各线程间共享内存和文件资源**，在线程之间数据传递的时候，就不需要经过内核。使得线程之间的**数据交互效率更高**了；[[线程间资源共享]]

## 线程的分类

根据线程最终实现的效果来进行分类的。实际上，用户级线程和内核级线程只是概念上的东西。

### 用户级线程

**用户级线程是指完全建立在用户空间的线程，对内核不可见**。用户级线程的建立，同步，销毁，调度完全在用户空间完成，不需要进行用户态和内核态之间的切换。**用户线程的整个线程管理和调度，操作系统是不直接参与的，而是由用户级线程库完成**。

> Python 中的线程是用户级线程

### 内核级线程

kernel-level threads

**内核级线程指的是内核可见的线程**。线程管理的所有工作由内核完成，应用程序没有进行线程管理的代码，只有一个到内核级线程的编程接口. 内核为进程及其内部的每个线程维护上下文信息，调度也是在内核基于线程架构的基础上完成。

**用户级线程和内核级线程的比较**

|                 | 用户级线程                                           | 内核级线程                                     |
| --------------- | ---------------------------------------------------- | ---------------------------------------------- |
| 线程切换        | **开销小**，线程切换直接在用户态中进行               | **开销大**，需要在用户态和内核态之间进行切换   |
| 线程阻塞        | **会影响其他线程**，因为用户级线程不受操作系统的调度 | **不会影响其他线程**，内核级线程受操作系统调度 |
| 抢占 CPU 时间片 | **不占优势**，在内核眼中所有用户线程共享一个调度单位 | **占优势**，在内核眼中是多个调度单位           |

## Linux 中线程实现相关概念

### ## Linux 中的线程实现：轻量级进程

轻量级进程本质上是内核级线程。

**在 Linux 中，并没有另外实现线程，而是使用轻量级（LWP）进程模拟了线程**。LWP 和进程一样也具有 PCB（`task_struct`），创建线程使用的底层函数和进程一样，都是`clone()`。调用 `clone()` 时使用不同的参数来指定创建 LWP 还是普通进程，通过参数决定子进程和父进程共享的资源种类和数量，这样就有了普通进程和轻量级进程的区别。在内核中，`clone()`调用经过参数传递和解释后会调用 `do_fork()`，这个核内函数同时也是 `fork()`、`vfork()`系统调用的最终实现。*Linux 中没有真正的线程，但 windows 中确实有线程*

LWP 与普通进程的区别也在于：**它只有一个最小的执行上下文和调度程序所需的统计信息**，而这也是它之所以被称为轻量级的原因。

### NPTL （Native POSIX Thread Library）


**在 Linux 2.6 以后的内核版本中，pthread 库开始由 NPTL 实现**。NPTL 并不是在 Linux 内核中加了一个“线程”，而是使用的 clone 实现的轻量级进程，内核又增加了若干机制来保证轻量级进程的表现和 POSIX 定义的线程相同，因此当然可以称它为“线程”。

因此在 Linux 2.6 的内核版本以后，用户调用 pthread 库创建的一个线程，会在内核创建一个“线程”，这就是所谓的 1：1 模型。所以，Linux 下，是有“内核级线程”的，网上很多说 Linux 是用户级线程，都是不完整的，说的 Linux 很早以前的版本（现在 Linux 已经是4.X 的版本了）。

pthread 线程函数库线程的底层实现决定了使用 pthread 线程函数库创建的线程是“用户级线程”还是“内核级线程”。它如果是用以前的方法，用它来创建的线程，是“用户级”线程；如果它使用了 NPTL 方式创建线程，它创建的线程，是“内核级”线程。

**因此使用 pthread 库时，只要：***
1. 使用 2.6 以后的内核的系统平台，
2. 2：你的 gcc 支持 NPTL （现在一般都支持），
那么编译出来的多线程程序，就是“内核级”线程了。

### 内核线程

kernel thread

Linux 内核可以看作一个服务进程(管理软硬件资源，响应用户进程的种种合理以及不合理的请求)。内核需要多个执行流并行，为了防止可能的阻塞，多线程化是必要的。

**内核线程就是内核的分身，一个分身可以处理一件特定事情**。Linux 内核使用内核线程来将内核分成几个功能模块，像 kswapd、kflushd 等，这在处理异步事件如异步 IO 时特别有用。内核线程的使用是廉价的，唯一使用的资源就是内核栈和上下文切换时保存寄存器的空间。**支持多线程的内核叫做多线程内核(Multi-Threads kernel )**。**内核线程的调度由内核负责，一个内核线程处于阻塞状态时不影响其他的内核线程，因为线程是调度的基本单位**。。


*线程的模型*

- 多对一 ： 用户级线程
- 一对一 ：内核级线程
- 多对多 ：用户级线程 + 内核级线程

## 线程设计模式

**Thread Pool (Boss/Worker)**

一个线程调度其他线程来执行有用的工作，这些工作通常是工作线程池的一部分。此线程池通常在 boss 线程开始调度线程工作之前预先分配。尽管线程是轻量级的，但它们在创建时仍会产生开销。

**Peer (Workcrew)**

Peer 模型类似于 boss/worker 模型，只是一旦创建了 worker 池，boss 就会成为线程池中的另一个线程，因此是其他线程的对等

**Pipline** 流水线

与处理器中的流水线工作方式类似，每个线程都是加工链中的一部分。每个线程处理由前一个线程处理的数据，并将其传递给下一个线程。必须小心均匀地分配工作并采取额外步骤来确保此线程模型中的非阻塞行为，否则您可能会遇到流水线中的“停滞(stalls)”。
