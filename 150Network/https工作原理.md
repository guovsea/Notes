HTTP 是明文传送，因此存在以下安全性问题：

**窃听风险** -- 加密 -- 混合加密：非对称加密（RSA 算法）[[非对称加密]] + 对称加密

> 非对称加密算法用于安全地交换对称加密的密钥，实际数据传输使用对称加密（性能更好）

**篡改风险** -- 校验 -- 摘要算法 + 数字签名[[数字签名]]

> 摘要算法可以计算出内容对应的哈希值，接收到内容后计算其哈希值并与发送方的哈希值进行比对。
> 但是*内容+哈希值*仍然可能被一同替换。

> 数字签名：用私钥对哈希值进行签名，保证内容+哈希值确实源于该公钥的所有者。
> 但是这个公钥本身可能也是伪造的。

**冒充风险** -- 身份证明 -- 数字证书[[数字证书]]

证明公钥不是被伪造的：
> 利用权威机构 CA（数字证书认证机构）的私钥对网站的公钥和网站的相关信息进行签名，得到数字证书，保证公钥不能被伪造。

## TLS 握手过程（TLS 1.2）

TLS 握手在 [[TCP三次握手]] 之后进行，目的是协商加密参数并建立安全通道：

1. **Client Hello**：客户端发送支持的 TLS 版本、密码套件列表、随机数（Client Random）
2. **Server Hello**：服务器选择 TLS 版本和密码套件，返回随机数（Server Random）+ [[数字证书]]
3. **密钥交换**：客户端验证证书有效性，生成预主密钥（Pre-Master Secret），用服务器公钥加密后发送
4. **生成会话密钥**：双方根据 Client Random + Server Random + Pre-Master Secret 计算出相同的会话密钥（对称密钥）
5. **完成**：双方用会话密钥加密完成消息，确认握手成功

> TLS 1.3 将握手简化为 1-RTT（甚至 0-RTT），移除了不安全的密码套件，安全性更高。

## TLS 1.2 vs TLS 1.3

| 特性 | TLS 1.2 | TLS 1.3 |
|------|---------|---------|
| 握手延迟 | 2-RTT | 1-RTT（支持 0-RTT 恢复） |
| 密码套件 | 支持 RSA 密钥交换、静态 DH | 仅支持 (EC)DHE，强制前向保密 |
| 安全性 | 存在已废弃的不安全算法 | 移除了所有已知不安全的算法 |
| 兼容性 | 广泛支持 | 现代浏览器均支持 |

> **前向保密（Forward Secrecy）**：即使服务器长期私钥泄露，历史通信记录也无法被解密。TLS 1.3 强制要求此特性。

## HTTPS 完整流程总结

```
客户端                                    服务器
  |                                         |
  |-------- TCP 三次握手 ------------------>|
  |                                         |
  |-------- TLS 握手 --------------------->|  ← 协商密钥、验证证书
  |                                         |
  |-------- HTTP 请求（加密传输） --------->|
  |<------- HTTP 响应（加密传输） ----------|
  |                                         |
  |-------- TCP 四次挥手 ------------------>|  ← [[断开TCP连接]]
```
