## 三次握手过程

```
客户端                                    服务器
  |                                         |
  |--- SYN, seq=x ----------------------->|   客户端: CLOSED → SYN_SENT
  |                                         |
  |<-- SYN+ACK, seq=y, ack=x+1 -----------|   服务器: LISTEN → SYN_RCVD
  |                                         |
  |--- ACK, ack=y+1 ---------------------->|   双方: ESTABLISHED
  |                                         |
```

- **第一次握手（SYN）**：客户端发送 SYN 报文，携带初始序列号 `seq=x`，进入 `SYN_SENT` 状态
- **第二次握手（SYN+ACK）**：服务器收到 SYN 后，回复 SYN+ACK 报文，确认号 `ack=x+1`，携带自己的初始序列号 `seq=y`，进入 `SYN_RCVD` 状态
- **第三次握手（ACK）**：客户端收到 SYN+ACK 后，发送 ACK 报文，`ack=y+1`，双方进入 `ESTABLISHED` 状态

> 第三次握手可以携带数据（RFC 793 允许），前两次不可以。

### 为什么 TCP 是三次握手而不是两次、四次握手

**1. 三次握手才可以阻止重复历史连接的初始化** *主要原因*

如果是两次握手的话，服务器端没有 `SYN_RCVD` 状态来阻止历史连接初始化（直接进入 `ESTABLISHED` 状态），导致服务器可能会建立一个历史连接（已失效的连接请求）。

**怎样避免历史连接的**

当服务器接收到一个历史连接的 SYN 报文时，会向客户端发送一个序列号为历史连接的序列号 + 1 的 `SYN+ACK` 报文并进入 `SYN_RCVD` 状态。客户端接收到这个报文后，**发现 ACK 序列号与当前新连接发送的 SYN 序列号并不匹配，便会向服务器发送 RST 报文**。服务器接收到 RST 报文后，不会进入 `ESTABLISHED` 状态，也就不会建立这个历史连接。


**2. 三次握手才能同步双方的初始序列号**

发送 SYN 后，只有收到对方回复的 ACK 才能确认对方已经收到自己的序列号。同理，对另一方来说也如此，因此四次握手也能同步双方初始序列号，只是将中间的"发送 ACK 和 SYN"合并为一次 `SYN+ACK` 了。

**3. 避免资源浪费**

如果只有「两次握手」，当客户端发送的 `SYN` 报文在网络中阻塞，客户端没有接收到 `ACK` 报文，就会重新发送 `SYN`，**由于没有第三次握手，服务端每收到一个 `SYN` 就只能先主动建立一个连接**，然而这些可能只是超时重传的 SYN，造成了服务器资源的浪费。

## SYN Flood 攻击与防御

攻击者发送大量伪造源 IP 的 SYN 报文，服务器为每个 SYN 分配资源进入 `SYN_RCVD` 状态，耗尽服务器资源。

防御手段：
- **SYN Cookie**：不在收到 SYN 时分配资源，而是将状态编码在 ACK 的序列号中，收到第三次 ACK 时才分配资源
- **限制半连接数**
- **防火墙过滤**
