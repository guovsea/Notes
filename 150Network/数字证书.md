## 作用

确定公钥不是被伪造的，解决 [[数字签名]] 中"公钥本身可能被伪造"的问题。

## 工作流程

1. **注册**：网站在 CA（数字证书认证机构）中注册并确认身份
2. **签发**：CA 会用自己的私钥对网站的信息（域名、组织等）和网站的公钥进行签名，得到数字证书
3. **验证**：客户端（浏览器）验证网站身份时，使用内置的 CA 公钥对数字证书进行解密/验证，得到网站的真实信息和公钥，确保公钥确实来源于该网站

## 证书的主要内容

一张 X.509 数字证书通常包含以下字段：

| 字段 | 说明 |
|------|------|
| 版本（Version） | 证书格式版本，通常为 v3 |
| 序列号（Serial Number） | CA 分配的唯一编号 |
| 签名算法 | CA 签名所使用的算法（如 SHA256withRSA） |
| 颁发者（Issuer） | 签发该证书的 CA 信息 |
| 有效期 | 证书的生效时间和过期时间 |
| 主体（Subject） | 证书持有者的信息（域名、组织等） |
| 公钥 | 持有者的公钥及算法 |
| 签名 | CA 对以上内容的 [[数字签名]] |

## 证书链

浏览器内置了根 CA 的公钥（根证书）。如果证书是由中间 CA 签发的，则需要通过证书链逐级验证，直到信任的根 CA。

```
根 CA 证书（内置在浏览器中）
  └── 中间 CA 证书（由根 CA 签发）
        └── 服务器证书（由中间 CA 签发）
```

> 为什么要用中间 CA？根 CA 的私钥极其重要，使用中间 CA 可以减少根 CA 私钥的使用频率，降低泄露风险；同时也方便按业务/地域分级管理。

## 证书类型

| 类型 | 验证级别 | 适用场景 |
|------|---------|---------|
| DV（Domain Validation） | 仅验证域名所有权 | 个人网站、博客 |
| OV（Organization Validation） | 验证组织/企业真实身份 | 企业官网、电商 |
| EV（Extended Validation） | 最严格的身份审核 | 银行、金融机构 |

## 证书吊销

当证书的私钥泄露或证书信息有误时，需要吊销证书：

- **CRL（证书吊销列表）**：CA 定期发布已被吊销的证书列表，客户端下载并检查。缺点是更新不及时、列表可能很大。
- **OCSP（在线证书状态协议）**：客户端实时向 CA 查询证书状态，响应更及时。OCSP Stapling 让服务器主动附带 OCSP 响应，避免客户端额外请求。
