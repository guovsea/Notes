## 基本原理

- 公钥加密，私钥解密 → 数据加密（[[非对称加密]]）
- **私钥加密，公钥解密 → 数字签名**（用于验证身份和数据完整性）

数字签名可以保证：
1. **身份认证**——确认消息确实来自声称的发送方（只有私钥持有者才能生成签名）
2. **数据完整性**——确认消息在传输过程中未被篡改
3. **不可否认性**——发送方无法否认自己发送过该消息

## 签名与验证过程

签名算法只对内容的哈希值进行签名（内容的哈希值通过摘要算法得到，如 SHA-256）。

> 为什么不对整个内容直接签名？因为非对称加密速度慢，对哈希值（固定长度，如 256 位）签名比对原始数据签名高效得多。

**签名过程**：`内容 → 哈希运算 → 哈希值 → 用私钥加密 → 数字签名`

**验证过程**：`数字签名 + 内容 + 公钥`

1. 首先利用公钥对数字签名进行解密，得到发送方的哈希值
2. 然后对接收到的内容计算实际的哈希值
3. 比较两个哈希值是否相同

**相同**：说明消息确实是被该公钥对应的私钥所签名的，内容和签名均有效

**不相同**：说明内容不来自公钥所有者，或内容已被篡改

> 解决的问题：在 [[https工作原理]] 中，仅靠摘要算法计算哈希值进行校验是不够的，因为内容和哈希值可能一同被中间人替换。数字签名利用私钥的唯一性，保证了内容确实来自声称的发送方。

## 常用签名算法

| 算法 | 说明 |
|------|------|
| RSA-PSS | 基于 RSA 的概率签名方案，安全性证明更完善 |
| ECDSA | 基于椭圆曲线，签名更短，性能更好 |
| EdDSA | 基于爱德华曲线（Ed25519），现代高效签名算法 |

## 应用场景

- **HTTPS/TLS**：服务器用私钥签名 TLS 握手信息，客户端验证身份
- **代码签名**：软件开发者对程序进行签名，系统验证来源可信
- **数字证书**：CA 机构用自己的私钥对证书进行签名（[[数字证书]]）
- **Git 提交签名**：开发者用 GPG 密钥签名 commit，证明提交者身份
