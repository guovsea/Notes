## 一、先确认你的定位：2 年经验的独特优势

2 年经验意味着你已经：
- 度过了最初的学习曲线，对至少一个技术栈比较熟练
- 有真实的代码审查和生产环境经验
- 理解团队协作和工程流程

这正是 **将 AI 作为杠杆放大器** 的最佳阶段 — 你有足够的基础知识来判断 AI 输出的质量，同时还有巨大的成长空间。

> "最好的 AI 辅助代码来自 **自己能写但选择用 AI 加速** 的工程师，而非不借助 AI 就写不出来的工程师。"

## 三、日常工作流：Spec → Plan → Act 三阶段法

这是 2026 年业界公认的最佳 AI 工作流：

### 阶段 1: 规划（Spec）— 你的核心价值所在

**不要直接跳到编码。** 先花 15 分钟写清楚要做什么：

```markdown
# Feature: 用户偏好 API

## 需求
- 支持获取/更新用户通知偏好
- 需要鉴权中间件
- 80% 测试覆盖率

## 技术上下文
- 技术栈: Spring Boot / PostgreSQL / Kafka
- 项目已有 UserRepository 和 AuthService

## 风险
- 并发更新可能的数据竞争
- 需要向后兼容旧版客户端

## 实现阶段
1. 设计数据模型和 API 契约
2. 实现 Controller + Service 层
3. 添加鉴权检查
4. 编写测试
```

**为什么这很重要：** 有了清晰的规格，AI 生成的代码准确率从 ~60% 提升到 ~90%。

### 阶段 2: 计划（Plan）— 让 AI 帮你拆解

在 Cursor Chat 或 Claude Code 中：

```
@codebase 分析现有代码结构，基于上面的规格，
列出需要修改的文件、依赖关系和实现顺序。
不要写代码，只输出计划。
```

审查 AI 的计划，确认它理解了：
- 现有架构模式
- 需要遵循的约定
- 潜在风险点

### 阶段 3: 执行（Act）— 小步迭代

**关键原则：不要一次性让 AI 实现整个功能。**

```
正确的做法：
Task 1 → 实现 + 验证 → Task 2 → 实现 + 验证 → ...

错误的做法：
"帮我实现整个用户偏好功能" → 得到500行代码 → 不知从何审查
```

## 四、四种模式的日常使用

### 模式 1: 自动补全（每天最频繁）

**场景：** 日常编码中的行级/函数级补全

```
你写: function formatDate(date: Date): string {
Copilot/Cursor 自动补全剩余实现

适合：
- 已知模式的样板代码
- 标准 CRUD 操作
- 常用工具函数
```

**注意：** 读一遍再接受，不要无脑 Tab。

### 模式 2: 对话式辅助（理解与学习）

**场景：** 不熟悉的代码/概念

```
@codebase 这个支付模块的错误处理逻辑是什么？
这段代码为什么要用 event sourcing 而不是直接 CRUD？
这个函数的并发安全吗？
```

**这是你成长最快的方式** — 用 AI 加速理解复杂系统，但要追问"为什么"。

### 模式 3: Agent 任务（复杂多文件变更）

**场景：** 涉及 3+ 文件的功能开发、重构

```
在 Claude Code 中：
"根据 SPEC.md 的描述，实现用户偏好 API。
 每完成一个步骤运行测试，确保不破坏现有功能。"
```

**你在做什么：** 监督 Agent 工作，审查每一步的变更。

### 模式 4: 审查模式（质量把关）

**场景：** 代码审查、安全检查

```
@codebase 审查我最近的变更：
- 是否有安全隐患？
- 是否遵循了项目的错误处理模式？
- 有没有遗漏的边界情况？
- 有没有过度工程化？
```

## 六、关键技巧：上下文工程

### 建立 CLAUDE.md（项目级 AI 指令文件）

在项目根目录创建 `CLAUDE.md`：

```markdown
# 项目约定

## 技术栈
- TypeScript + React + Next.js
- Prisma ORM + PostgreSQL
- Jest + Playwright 测试

## 编码规范
- 使用函数式组件，禁止 class 组件
- API 路由统一用 try-catch 包裹
- 错误码遵循 RFC 7807 格式
- 禁止使用 any 类型

## 关键架构决策
- 认证使用 JWT + Refresh Token
- 支付模块不使用 AI 生成代码（手动编写）
- 数据库迁移必须经过人工审查

## 文件结构
- /src/app — Next.js 路由
- /src/services — 业务逻辑层
- /src/repositories — 数据访问层
```

> 这个文件就像给 AI 的一份"入职文档"。开发者反馈：建立 CLAUDE.md 后，AI 输出质量提升显著。

### 给 AI 正确的上下文

```
❌ 差的上下文:
"帮我写一个登录功能"

✅ 好的上下文:
@/src/services/authService.ts @/src/middleware/auth.ts
项目使用 JWT 认证，需要：
1. 接收 email + password
2. 校验后返回 access_token(15min) + refresh_token(7d)
3. 失败返回 401 + RFC 7807 错误格式
4. 写对应的 Jest 测试
```

## 七、必须避免的 6 个陷阱

### 陷阱 1: 无脑接受 AI 建议

> "我审查了 50 个 AI 生成的 PR，发现代码在**它想象的世界里**几乎总是正确的。问题是那个世界不存在。"

**AI 写代码时的隐含假设：**
- 每个外部服务都会正常响应
- 数据库查询总是返回期望的结果
- 不会有并发问题
- 用户会按预期使用

**对策：** 审查时重点看 **错误处理** 和 **边界情况**，而非正常路径。

### 陷阱 2: 一次性大任务

```
❌ "帮我重构整个认证模块"
→ 500行代码，审查困难，容易出问题

✅ "第一步：将 JWT secret 移到环境变量"
→ 20行变更，容易验证
→ "第二步：添加 token 过期检查"
→ 15行变更，容易验证
```

### 陷阱 3: 技能退化

2 年经验正是打基础的关键期。建议：

- **每周至少 2 次手动编码时段**（不使用 AI 补全）
- 用 AI 生成代码后，**尝试从记忆重写核心逻辑**
- 遇到 AI 建议不熟悉的模式时，**追问"为什么用这个方式"**
- 自己维护一个"AI 解决不了的问题"笔记本

### 陷阱 4: 盲目引入新依赖

> "AI 生成的代码频繁引入新依赖。一次审计中我发现一个团队增加了 23 个新 npm 包，其中 15 个是多余的。"

**对策：** 审查每个 `import` — 这个库项目里已经有了吗？真的需要吗？

### 陷阱 5: 跳过测试

AI 生成的代码看起来正确，但：

```
实际数据：
├── AI 生成代码的 Bug 密度: 比人写的高 1.7x
├── 逻辑错误: 增加 75%
├── 性能问题: 增加 8x
└── 安全漏洞: 增加 1.5-2x
```

**对策：** 让 AI **先写测试，再写实现**（TDD with AI）：
```
1. 你描述需求和边界条件
2. AI 生成测试用例（你来审核补充）
3. AI 实现代码让测试通过
4. 你审查最终代码
```

### 陷阱 6: 让 AI 做架构决策

> "AI 非常擅长实现模式，但非常不擅长选择用哪个模式。"

**AI 做**：实现具体的代码、写测试、重构、文档
**你做**：选择架构模式、决定技术方案、权衡取舍

## 九、核心心法总结

```
┌─────────────────────────────────────────────────────┐
│                                                      │
│   1. 你是架构师，AI 是执行者                           │
│      你决定做什么，AI 帮你怎么做                        │
│                                                      │
│   2. 先规格，后编码                                    │
│      15 分钟的规划节省数小时的返工                      │
│                                                      │
│   3. 小步快跑，逐步验证                                │
│      每次只让 AI 做一件事，做完验证再做下一件             │
│                                                      │
│   4. 审查 AI 代码比审查人写的代码要更严格               │
│      重点看：错误处理、边界情况、假设合理性               │
│                                                      │
│   5. 保持手动编码能力                                  │
│      这是你在 3AM 生产事故时唯一能依赖的东西              │
│                                                      │
│   6. 用 AI 学习，而非逃避学习                          │
│      "我理解 AI 做了什么" vs "我跳过了理解"             │
│                                                      │
└─────────────────────────────────────────────────────┘
```

> **最终目标：** 不是成为"最会用 AI 的人"，而是成为 **有扎实基础、又善用 AI 杠杆** 的工程师。AI 让编码变快了，但你的判断力、系统思维和领域知识才是真正的护城河。
